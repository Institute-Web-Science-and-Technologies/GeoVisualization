\subsubsection*{Server}
%Der implementierte Server arbeitet mit Java und ZeroMQ. Er öffnet zwei Kommunikationskanäle mit dem Client. Der erste ist ein ZeroMQ-Request-Reply-Socket-Paar, über das die Endgeräte Nachrichten an den Server senden. Der zweite ist ein Publish-Subscribe-Socket-Paar, über das die Nachrichten an Gruppen von Endgeräten weitergeleitet werden. Eine Nachricht besteht aus drei Teilen: einer Adresse, dem Nachrichtentyp und einem serialisierten Objekt vom Typ TransferObject (siehe Abbildung \ref{fig:transfer}). Die Adresse ist entweder die ID eines Spielers oder die ID einer Spielinstanz. Wir nutzen ZeroMQs Multipart-Message-Feature um diese Teile voneinander getrennt bei der Kommunikation zu übermitteln. Der Server sendet eingehende Nachrichten an bestimmte Clients weiter. Das kann entweder ein einzelner Client, oder alle Clients die sich in einer Spielsession befinden, sein. Der Server kennt dabei den Zustand der einzelnen Spiele nicht. , Um mehrere Spielinstanzen gleichzeitig verwalten zu können, hält er aber eine Liste aller laufenden Spiele samt Informationen darüber, welcher Spieler an welchem Spiel teilnimmt. Wenn eine Nachricht vom Typ \glqq create\_game\grqq empfangen wird trägt der Server die ID des Spiels in eine Liste ein. Diese Liste wird einem Client gesendet wenn er sie über eine entsprechende Nachricht anfragt.

\paragraph{Kommunikation}
Im Hintergrund laufen zwei Threads für die Kommunikation. Der eine kümmert sich um den Empfang von Nachrichten und beinhaltet einen ZeroMQ-Subscriber-Socket. Dieser empfängt alle Nachrichten, die vom Publisher"=Socket auf dem Server direkt an die ID des Clients oder die ID des Spiels, in dem er sich befindet, adressiert sind. Der zweite Thread sendet Nachrichten an den Server. 

\paragraph{Darstellung der Schlange}
Die Schlange wird als Polygonzug (eine Menge von Punkten zwischen denen Linien gezeichnet werden) dargestellt. Hierfür wird eine Polyline \footnote{\url{https://developers.google.com/android/reference/com/google/android/gms/maps/model/Polyline}} verwendet. Der Polygonzug hat jeweils nur eine bestimmte Länge, die von den erreichten Punkten des Spielers abhängt. Es wird jeweils die neue Position des Spielers am Anfang des Polygonzugs (Kopf)  eingetragen. Entsprechend der erlaubten Länge, werden die letzten Punkte gelöscht.

\paragraph{Objekte aufsammeln}
Jeder Client überprüft nur für sich selbst ob er nahe genug an einem Objekt ist um diesen aufzusammeln und sendet diese Information weiter an alle Mitspieler inklusive sich selbst. Erst beim Empfang dieser Nachricht wird die Länge der Schlange verändert und der Zielpunkt (das Item) von der Karte entfernt.  Die Zielpunkte werden zufällig in der Nähe von Spielern erzeugt falls die Anzahl an momentan im Spiel befindlichen Punkte einen einstellbaren Wert unterschreitet.

\paragraph{Kollision}
Die Kollision wird über Abstandsmessung realisiert (s. \ref{abstandsmessung}). Hierbei wird bei jedem Positionsupdate die Distanz zu allen kollisionsrelevanten Punkten (Eigener Schwanz, komplette Schlangen anderer Spieler) gemessen. Unterschreitet die Distanz einen gewissen Wert, wird eine Kollision ausgelöst. Aufgrund der schwankenden Positionsupdatefrequenz und Bewegungsgeschwindigkeit der Spieler, wird bei der Eigenkollision in einem vorher festgelegter Radius um den Kopf der Schlange die Kollision mit dem eigenen Schwanz ignoriert. Im Falle des Spielstartes oder "`Tod"' der Schlange wird jegliche Kollision für einen kurzen Zeitraum deaktiviert.
